# 清理历史遗留的知识库孤儿数据

LobeHub 的较新版本已经修复了知识库删除链路，使未来的知识库删除操作能够同步清理其独占文件、向量数据与对象存储对象。然而，在修复上线之前已经积累下来的历史数据，仍然可能保留残留数据。

本文档给出一套保守的运维方案，用于识别并清理这些历史残留，同时尽量避免额外的数据误删风险。

## 修复前究竟泄漏了什么

| 层级    | 主要表或对象                                                | 历史行为                         | 空间影响                |
| ----- | ----------------------------------------------------- | ---------------------------- | ------------------- |
| 知识库关联 | `knowledge_bases`、`knowledge_base_files`              | 删除知识库时，关联行被删掉，但其背后的文件行没有同步删除 | 容器消失了，但真实载荷仍在       |
| 文件元数据 | `files`、`global_files`                                | 文件记录会在知识库删除后继续保留             | 文件配额与对象元数据继续占用空间    |
| 向量链路  | `file_chunks`、`chunks`、`embeddings`、`document_chunks` | 由于文件行仍然存在，分块与向量行也会继续保留       | PGVector 与关系型存储持续增长 |
| 对象存储  | S3 兼容存储中的对象                                           | 原始上传对象不会自动回收                 | 直到对象被删除前，存储费用仍持续计费  |

```text
[修复前删除知识库]
          |
          v
[knowledge_bases 行被删除]
          |
          v
[knowledge_base_files 级联删除]
          |
          v
[files 行仍然保留]
          |
          v
[file_chunks -> chunks -> embeddings 仍然保留]
          |
          v
[S3 对象仍然保留]
```

- 修复后的删除链路可以阻止后续继续累积这类残留。
- 历史残留之所以更难处理，是因为原始知识库上下文已经被删掉。
- 因此，治理动作必须从 “报表与抽样” 开始，而不是直接进行硬删除。

## 核心约束

| 约束                              | 影响                                                      |
| ------------------------------- | ------------------------------------------------------- |
| 历史归属信息已经丢失                      | `knowledge_base_files` 级联删除后，数据库已无法直接证明某个文件原本属于哪个已删除知识库 |
| 文件可能被知识库之外的功能复用                 | 同一文件可能还被消息、Agent、会话、文档、TTS 或生成记录引用                      |
| 对象存储 URL 的形态不一定统一               | 有的部署保存对象 key，有的旧数据可能保存完整 URL                            |
| PostgreSQL 的 “删行” 和 “缩盘” 不是同一件事 | 先释放逻辑占用，操作系统层面的磁盘回收通常还需要 vacuum 或 repack                |

- 这意味着不存在一个完全无风险、完全精准的 “历史全量清理 SQL”。
- 正确策略应当是：保守候选集、人工抽样确认、分批删除、逐批验证。

## 推荐执行流程

| 阶段 | 动作                                      | 预期产出                              |
| -- | --------------------------------------- | --------------------------------- |
| 1  | 先升级到已包含修复的版本                            | 清理窗口内不再继续产生新的历史残留                 |
| 2  | 备份 PostgreSQL，并尽可能导出 S3 inventory 或对象清单 | 获得破坏性操作前的恢复点                      |
| 3  | 物化一份保守的候选报表                             | 先得到可审阅的数据集合，而不是直接删除               |
| 4  | 抽样检查高风险用户、最旧数据、向量占用最高的数据                | 验证筛选条件不会误伤活跃数据                    |
| 5  | 优先通过应用层代码分批删除                           | 数据库行、向量行与对象存储对象可以按既有链路一致清理        |
| 6  | 做验证与后续维护                                | 逻辑用量立即下降，PostgreSQL 空间可复用，并可进一步整理 |

- 生产环境建议采用小批量执行。
- 建议保留至少七天的年龄缓冲，避免与近期上传或后台任务发生竞争。
- 若部署有审计要求，应在删除前导出候选集。

## 保守候选报表

下面的查询会刻意排除仍被其他产品面引用的文件。它并不试图 “还原最初归属”，而是找出那些已经脱离主要活跃引用面的文件，因此适合作为第一轮清理候选。

```sql
CREATE TEMP TABLE maintenance_kb_orphan_candidates AS
WITH active_refs AS (
  SELECT file_id FROM knowledge_base_files
  UNION
  SELECT file_id FROM messages_files
  UNION
  SELECT file_id FROM agents_files
  UNION
  SELECT file_id FROM files_to_sessions
  UNION
  SELECT file_id FROM generations WHERE file_id IS NOT NULL
  UNION
  SELECT file_id FROM message_tts WHERE file_id IS NOT NULL
  UNION
  SELECT file_id FROM documents WHERE file_id IS NOT NULL
),
vector_usage AS (
  SELECT
    fc.file_id,
    COUNT(DISTINCT fc.chunk_id) AS chunk_count,
    COUNT(DISTINCT e.id) AS embedding_count
  FROM file_chunks fc
  LEFT JOIN embeddings e ON e.chunk_id = fc.chunk_id
  GROUP BY fc.file_id
)
SELECT
  f.id,
  f.user_id,
  f.name,
  f.file_hash,
  f.url,
  f.size,
  f.created_at,
  COALESCE(v.chunk_count, 0) AS chunk_count,
  COALESCE(v.embedding_count, 0) AS embedding_count
FROM files f
LEFT JOIN active_refs r ON r.file_id = f.id
LEFT JOIN vector_usage v ON v.file_id = f.id
WHERE r.file_id IS NULL
  AND f.parent_id IS NULL
  AND f.created_at < NOW() - INTERVAL '7 days';
```

按用户汇总查看：

```sql
SELECT
  user_id,
  COUNT(*) AS file_count,
  SUM(size) AS total_bytes,
  SUM(chunk_count) AS total_chunks,
  SUM(embedding_count) AS total_embeddings
FROM maintenance_kb_orphan_candidates
GROUP BY user_id
ORDER BY total_embeddings DESC, total_bytes DESC;
```

如果只想先处理带向量占用的候选文件，可以在第一段查询末尾追加：

```sql
AND COALESCE(v.chunk_count, 0) > 0
```

- 这组条件是 “保守识别”，而不是 “穷尽识别”。
- 它适合作为第一轮生产治理。
- 如果你的部署语义明确要求扩大清理范围，应在完成抽样确认后再放宽条件。

## 推荐删除路径

推荐的实现方式是编写一次性维护脚本，复用当前修复后的应用层删除逻辑，而不是重新发明一条新的删除链路。

```text
[候选文件行]
      |
      v
[按 user_id 分组]
      |
      v
[FileModel.deleteMany(fileIds)]
      |
      +--> 删除 files 行
      +--> 删除 file_chunks
      +--> 删除 chunks
      +--> 删除 embeddings
      +--> 删除无剩余引用的 global_files
      |
      v
[必要时先规范化对象存储 key]
      |
      v
[FileService.deleteFiles(keys)]
```

维护脚本骨架可参考：

```ts
for (const [userId, rows] of candidatesGroupedByUser) {
  const fileModel = new FileModel(db, userId);
  const fileService = new FileService(db, userId);

  for (const batch of chunk(rows, 200)) {
    const deleted = await fileModel.deleteMany(
      batch.map((row) => row.id),
      true,
    );

    const keys = (
      await Promise.all(
        deleted
          .map((row) => row.url)
          .filter((url): url is string => Boolean(url))
          .map(async (url) => {
            if (url.startsWith('http://') || url.startsWith('https://')) {
              return await fileService.getKeyFromFullUrl(url);
            }

            return url;
          }),
      )
    ).filter((key): key is string => Boolean(key));

    if (keys.length > 0) {
      await fileService.deleteFiles(keys);
    }
  }
}
```

- `FileModel.deleteMany(...)` 是关键，因为它会按照现有应用约定删除文件相关的向量链路数据。
- 如果历史 `files.url` 保存的是完整 URL 而不是对象 key，则在调用 `deleteFiles(...)` 前必须先做归一化。
- 批量大小应根据数据库体量、锁容忍度与 S3 速率限制来决定。

## 仅用 SQL 的回退方案

如果你的环境无法直接执行应用代码，也可以采用纯 SQL 回退方案。该方案可操作，但需要你单独完成对象存储清理，因此整体体验不如应用层脚本。

在删除前，先通过 SQL 客户端导出候选集，保留 `id`、`user_id`、`file_hash`、`url` 等字段，用于审计与对象存储删除。

```sql
BEGIN;

CREATE TEMP TABLE maintenance_kb_orphan_chunk_ids AS
SELECT DISTINCT fc.chunk_id
FROM file_chunks fc
JOIN maintenance_kb_orphan_candidates c ON c.id = fc.file_id;

DELETE FROM embeddings e
USING maintenance_kb_orphan_chunk_ids x
WHERE e.chunk_id = x.chunk_id;

DELETE FROM document_chunks dc
USING maintenance_kb_orphan_chunk_ids x
WHERE dc.chunk_id = x.chunk_id;

DELETE FROM chunks c
USING maintenance_kb_orphan_chunk_ids x
WHERE c.id = x.chunk_id;

DELETE FROM files f
USING maintenance_kb_orphan_candidates c
WHERE f.id = c.id;

DELETE FROM global_files gf
WHERE NOT EXISTS (
  SELECT 1
  FROM files f
  WHERE f.file_hash = gf.hash_id
);

COMMIT;
```

- 删除 `chunks` 时，依赖 `ON DELETE CASCADE` 的关联表会一并级联清理。
- S3 兼容对象存储中的物理对象仍需独立删除。
- 如果历史数据里存在完整 URL，请先将其规范化为对象 key，再执行桶对象删除。

## 哪些动作才真正释放空间

| 空间类型                      | 何时释放                                    | 补充说明                          |
| ------------------------- | --------------------------------------- | ----------------------------- |
| 用户侧可见的文件用量与向量计数           | 成功删行后立即下降                               | 应用层统计会最先回落                    |
| PostgreSQL 可复用表空间         | 删除后经过 autovacuum 或手动 `VACUUM (ANALYZE)` | PostgreSQL 可以复用，但不一定立即归还给操作系统 |
| PostgreSQL 在操作系统层面的实际文件大小 | 通常需要 `pg_repack` 或 `VACUUM FULL`        | `VACUUM FULL` 锁更重，应安排维护窗口     |
| S3 兼容对象存储计费               | 对象删除完成后                                 | 若桶启用了版本控制，还需考虑历史版本清理          |

建议在清理后执行：

```sql
VACUUM (ANALYZE) files;
VACUUM (ANALYZE) file_chunks;
VACUUM (ANALYZE) chunks;
VACUUM (ANALYZE) embeddings;
VACUUM (ANALYZE) global_files;
```

- `VACUUM (ANALYZE)` 适合在分批删除后做常规整理。
- 如果目标是尽快缩小操作系统层面的数据库文件大小，优先考虑 `pg_repack`。
- `VACUUM FULL` 应仅在可控维护窗口中执行。

## 验证清单

| 检查项                    | 查询或方法                                          | 预期结果          |
| ---------------------- | ---------------------------------------------- | ------------- |
| 候选集已人工审阅               | 查看高风险用户与随机样本                                   | 不应出现明显仍在使用的文件 |
| 文件行已删除                 | `SELECT COUNT(*) FROM files WHERE id IN (...)` | `0`           |
| 向量行已删除                 | 基于候选 `chunk_id` 回查                             | `0`           |
| 无引用 `global_files` 已清理 | 用 `global_files` 对剩余 `files.file_hash` 回查      | 不再存在陈旧元数据     |
| 桶对象已删除                 | 通过 S3 inventory、CLI 或控制台确认                     | 对应对象已消失       |
| 数据库统计已刷新               | `VACUUM (ANALYZE)` 完成                          | 查询计划与统计信息已更新  |

- 应逐批验证，而不是等全部批次结束后再统一检查。
- 建议保留候选集导出文件与批次日志，便于回滚分析。
- 如果后续需要沉淀成正式管理命令，建议直接构建在 `FileModel` 与 `FileService` 之上，而不是再实现一套新的删除路径。
