# Cleaning Historical Orphaned Knowledge Base Data

Recent versions of LobeHub fix the deletion path for knowledge bases so that future deletions can also remove exclusively owned files, vector data, and object-storage artifacts. However, deployments that accumulated data before this fix may still retain historical residual data.

This document describes a conservative operational procedure for identifying and removing that historical residue without introducing avoidable data loss.

## What Leaked Before the Fix

| Layer                  | Main tables or objects                                   | Historical behavior                                                                   | Space impact                                                 |
| ---------------------- | -------------------------------------------------------- | ------------------------------------------------------------------------------------- | ------------------------------------------------------------ |
| Knowledge base linkage | `knowledge_bases`, `knowledge_base_files`                | Deleting a knowledge base removed the linkage rows, but not the file rows behind them | The logical container disappeared, but payload data remained |
| File metadata          | `files`, `global_files`                                  | File records could remain after the knowledge base was deleted                        | File usage and deduplicated object metadata stayed allocated |
| Vector pipeline        | `file_chunks`, `chunks`, `embeddings`, `document_chunks` | Chunk and embedding rows could remain reachable through surviving file rows           | PGVector and relational storage continued to grow            |
| Object storage         | S3-compatible bucket objects                             | Raw uploaded objects were not reclaimed automatically                                 | Storage billing continued until object deletion              |

```text
[Delete knowledge base before fix]
            |
            v
[knowledge_bases row deleted]
            |
            v
[knowledge_base_files rows cascade away]
            |
            v
[files rows remain]
            |
            v
[file_chunks -> chunks -> embeddings remain]
            |
            v
[S3 object remains]
```

- The fixed code path prevents this accumulation for future deletions.
- Historical residue is more difficult because the original knowledge base context has already been removed.
- Consequently, remediation must begin with reporting and sampling, not immediate hard deletion.

## Operational Constraint

| Constraint                                                   | Why it matters                                                                                                                        |
| ------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------- |
| Historical provenance has already been lost                  | Once `knowledge_base_files` rows cascaded away, the database no longer records which deleted knowledge base originally owned the file |
| Files can be reused outside knowledge bases                  | A file may also be referenced by messages, agents, sessions, documents, TTS, or generated assets                                      |
| Object-storage URLs may not be normalized                    | Some deployments store object keys directly, while older records may contain full URLs                                                |
| PostgreSQL row deletion and disk shrink are different events | Row deletion releases logical usage first; operating-system disk shrink usually requires vacuum or repack work                        |

- The practical consequence is that there is no perfectly lossless retrospective cleanup query.
- The correct strategy is therefore a conservative candidate filter followed by manual review and batch deletion.

## Recommended Runbook

| Phase | Action                                                                        | Expected output                                                                               |
| ----- | ----------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
| 1     | Upgrade to a version that includes the knowledge-base deletion fix            | No new historical residue is created during the cleanup window                                |
| 2     | Take a PostgreSQL backup and, if possible, an S3 inventory or bucket snapshot | Recovery point before destructive operations                                                  |
| 3     | Materialize a conservative candidate report                                   | A reviewable candidate set rather than direct deletion                                        |
| 4     | Review top users, oldest rows, and highest-vector rows                        | Confidence that the filter is not removing active user data                                   |
| 5     | Delete in batches through application code when possible                      | Database rows, vector rows, and object-storage objects are cleaned consistently               |
| 6     | Run verification and maintenance work                                         | Logical usage drops immediately; PostgreSQL space becomes reusable and can later be compacted |

- For production systems, perform the deletion phase in small batches.
- A seven-day age buffer is recommended to avoid racing recent uploads or background jobs.
- If the deployment has strict audit requirements, export the candidate set before deletion.

## Conservative Candidate Report

The following report intentionally excludes files that are still referenced by other product surfaces. It does not attempt to reconstruct original ownership. Instead, it identifies files that are detached from all major active references and are therefore strong cleanup candidates.

```sql
CREATE TEMP TABLE maintenance_kb_orphan_candidates AS
WITH active_refs AS (
  SELECT file_id FROM knowledge_base_files
  UNION
  SELECT file_id FROM messages_files
  UNION
  SELECT file_id FROM agents_files
  UNION
  SELECT file_id FROM files_to_sessions
  UNION
  SELECT file_id FROM generations WHERE file_id IS NOT NULL
  UNION
  SELECT file_id FROM message_tts WHERE file_id IS NOT NULL
  UNION
  SELECT file_id FROM documents WHERE file_id IS NOT NULL
),
vector_usage AS (
  SELECT
    fc.file_id,
    COUNT(DISTINCT fc.chunk_id) AS chunk_count,
    COUNT(DISTINCT e.id) AS embedding_count
  FROM file_chunks fc
  LEFT JOIN embeddings e ON e.chunk_id = fc.chunk_id
  GROUP BY fc.file_id
)
SELECT
  f.id,
  f.user_id,
  f.name,
  f.file_hash,
  f.url,
  f.size,
  f.created_at,
  COALESCE(v.chunk_count, 0) AS chunk_count,
  COALESCE(v.embedding_count, 0) AS embedding_count
FROM files f
LEFT JOIN active_refs r ON r.file_id = f.id
LEFT JOIN vector_usage v ON v.file_id = f.id
WHERE r.file_id IS NULL
  AND f.parent_id IS NULL
  AND f.created_at < NOW() - INTERVAL '7 days';
```

To inspect the result set by user:

```sql
SELECT
  user_id,
  COUNT(*) AS file_count,
  SUM(size) AS total_bytes,
  SUM(chunk_count) AS total_chunks,
  SUM(embedding_count) AS total_embeddings
FROM maintenance_kb_orphan_candidates
GROUP BY user_id
ORDER BY total_embeddings DESC, total_bytes DESC;
```

To focus only on vector-bearing candidates, add the following predicate to the first query:

```sql
AND COALESCE(v.chunk_count, 0) > 0
```

- This filter is intentionally conservative rather than exhaustive.
- It is appropriate for the first cleanup pass.
- If your deployment semantics guarantee that more detached files should be removed, widen the predicate only after sampling.

## Preferred Deletion Path

The preferred implementation is a one-off maintenance script that reuses the same application-layer deletion logic now used by the fixed knowledge-base flow.

```text
[Candidate rows]
      |
      v
[Group by user_id]
      |
      v
[FileModel.deleteMany(fileIds)]
      |
      +--> deletes file rows
      +--> deletes file_chunks
      +--> deletes chunks
      +--> deletes embeddings
      +--> deletes unreferenced global_files
      |
      v
[Normalize object-storage keys if needed]
      |
      v
[FileService.deleteFiles(keys)]
```

Suggested maintenance-script skeleton:

```ts
for (const [userId, rows] of candidatesGroupedByUser) {
  const fileModel = new FileModel(db, userId);
  const fileService = new FileService(db, userId);

  for (const batch of chunk(rows, 200)) {
    const deleted = await fileModel.deleteMany(
      batch.map((row) => row.id),
      true,
    );

    const keys = (
      await Promise.all(
        deleted
          .map((row) => row.url)
          .filter((url): url is string => Boolean(url))
          .map(async (url) => {
            if (url.startsWith('http://') || url.startsWith('https://')) {
              return await fileService.getKeyFromFullUrl(url);
            }

            return url;
          }),
      )
    ).filter((key): key is string => Boolean(key));

    if (keys.length > 0) {
      await fileService.deleteFiles(keys);
    }
  }
}
```

- `FileModel.deleteMany(...)` is the important part because it removes vector-related rows in the correct application order.
- If your historical `files.url` values are full URLs rather than object keys, normalize them before calling `deleteFiles(...)`.
- Batch size should be chosen according to database size, lock tolerance, and S3 rate limits.

## SQL-Only Fallback

If application code cannot be executed in your environment, a SQL-only fallback is possible. This approach is less ergonomic and requires separate object-storage cleanup, but it is still operationally viable.

Before deletion, export the candidate list with your SQL client so that you retain `id`, `user_id`, `file_hash`, and `url` for audit and storage cleanup.

```sql
BEGIN;

CREATE TEMP TABLE maintenance_kb_orphan_chunk_ids AS
SELECT DISTINCT fc.chunk_id
FROM file_chunks fc
JOIN maintenance_kb_orphan_candidates c ON c.id = fc.file_id;

DELETE FROM embeddings e
USING maintenance_kb_orphan_chunk_ids x
WHERE e.chunk_id = x.chunk_id;

DELETE FROM document_chunks dc
USING maintenance_kb_orphan_chunk_ids x
WHERE dc.chunk_id = x.chunk_id;

DELETE FROM chunks c
USING maintenance_kb_orphan_chunk_ids x
WHERE c.id = x.chunk_id;

DELETE FROM files f
USING maintenance_kb_orphan_candidates c
WHERE f.id = c.id;

DELETE FROM global_files gf
WHERE NOT EXISTS (
  SELECT 1
  FROM files f
  WHERE f.file_hash = gf.hash_id
);

COMMIT;
```

- Deleting from `chunks` will cascade to relations such as `file_chunks` and any other chunk-linked association tables that use `ON DELETE CASCADE`.
- Physical objects in S3-compatible storage must still be deleted separately.
- If you use this fallback, normalize legacy full URLs into object keys before issuing bucket deletions.

## What Actually Releases Space

| Space type                                         | When it is released                                                   | Additional note                                                         |
| -------------------------------------------------- | --------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| User-visible file usage and vector counts          | Immediately after successful row deletion                             | Application-level storage accounting should drop first                  |
| PostgreSQL reusable table space                    | After normal delete plus autovacuum or manual `VACUUM (ANALYZE)`      | Reusable by PostgreSQL, but not always returned to the operating system |
| PostgreSQL file size at the operating-system level | Usually after `pg_repack` or `VACUUM FULL`                            | `VACUUM FULL` takes stronger locks and should be scheduled carefully    |
| S3-compatible object storage billing               | After object deletion completes and lifecycle/versioning rules settle | Versioned buckets may require additional noncurrent-version cleanup     |

Recommended post-cleanup maintenance:

```sql
VACUUM (ANALYZE) files;
VACUUM (ANALYZE) file_chunks;
VACUUM (ANALYZE) chunks;
VACUUM (ANALYZE) embeddings;
VACUUM (ANALYZE) global_files;
```

- `VACUUM (ANALYZE)` is suitable for routine cleanup after batched deletes.
- If the objective is immediate operating-system disk shrink, prefer `pg_repack` when available.
- Reserve `VACUUM FULL` for a controlled maintenance window.

## Verification Checklist

| Check                             | Query or method                                          | Expected outcome                      |
| --------------------------------- | -------------------------------------------------------- | ------------------------------------- |
| Candidate table reviewed          | Inspect top users and random samples                     | No obviously active files are present |
| File rows deleted                 | `SELECT COUNT(*) FROM files WHERE id IN (...)`           | `0`                                   |
| Vector rows deleted               | Join through previous candidate chunk IDs                | `0`                                   |
| Unreferenced global files removed | Query `global_files` against remaining `files.file_hash` | No stale metadata rows remain         |
| Bucket objects deleted            | S3 inventory, CLI, or provider console                   | Matching objects are gone             |
| Database statistics refreshed     | `VACUUM (ANALYZE)` completed                             | Query plans and counters are current  |

- Perform verification after each batch, not only at the end of the full run.
- Store the candidate export and batch logs for rollback analysis.
- If you need a reusable admin command, build it on top of the same `FileModel` and `FileService` methods rather than inventing a second deletion path.
